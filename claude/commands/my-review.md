現在の修正内容をテックリード視点でレビューする。

## 引数

- 引数なし: `git diff master...HEAD` で現ブランチ全体を対象にする
- 引数あり: `git diff $ARGUMENTS` で任意の比較を対象にする（例: `HEAD~1`, `main..feature/xxx`）

## 手順

1. 引数に応じて diff を取得する。
   - 引数なし: `git diff master...HEAD`
   - 引数あり: `git diff $ARGUMENTS`
2. diff が空の場合は「変更なし」と報告して終了する。
3. 以下の観点でレビューを行う。

### レビュー観点

1. **要件適合** — 変更が意図した目的を正確に達成しているか
2. **副作用** — 既存機能への影響・破壊的変更がないか
3. **シンプルさ** — 不要な抽象化・冗長なコード・共通化できる処理がないか
4. **エッジケース** — 境界値・null・空配列・異常系の考慮漏れがないか
5. **セキュリティ** — インジェクション・認証漏れ・機密情報の露出がないか
6. **命名・可読性** — 変数名・関数名が意図を正確に表しているか

## 出力フォーマット

```
## レビュー結果

**対象**: <diff の比較範囲>
**変更ファイル数**: N ファイル

### 問題あり

- [重要度: 高/中/低] `ファイル名:行番号` — 問題の説明と修正提案

### 問題なし

- 観点名: OK

### 総評

<全体的な評価と次のアクション>
```

問題が見つかった場合は修正を提案するだけでなく、ユーザーの許可を得てから実際に修正すること。
